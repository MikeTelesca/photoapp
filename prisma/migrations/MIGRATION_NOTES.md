# Migration Catch-Up Notes

## Background

`prisma db push` was used during development to avoid disrupting the Neon Postgres production database.
As a result, Prisma's `_prisma_migrations` history table is empty (or doesn't exist), even though the
live schema may be partially in sync. These files capture **all** DDL needed to bring prod fully in sync
with `prisma/schema.prisma` as of 2026-04-14.

## Files in this directory

| File | Description |
|---|---|
| `auto-diff.sql` | Generated by `prisma migrate diff --from-empty --to-schema` — canonical Prisma output |
| `manual-catchup.sql` | Hand-written idempotent version using `IF NOT EXISTS` — safe to run multiple times |

**Recommendation:** Use `auto-diff.sql` if the prod DB has **none** of these tables yet (fresh Neon
branch). Use `manual-catchup.sql` if prod already has some tables and you need a safe incremental apply.

---

## How to apply to production

### Option A — psql direct (recommended)

```bash
# Connect to your Neon prod DB
psql "$DATABASE_URL" -f prisma/migrations/auto-diff.sql
# OR for idempotent apply:
psql "$DATABASE_URL" -f prisma/migrations/manual-catchup.sql
```

### Option B — Neon console SQL editor

Paste the contents of either file into the Neon SQL Editor and run.

### After applying

Tell Prisma the migration has been applied so it stops showing drift warnings:

```bash
# Mark the migration as applied in _prisma_migrations
npx prisma migrate resolve --applied 20260414000000_manual_catchup
```

*(The timestamp value is arbitrary — it just needs to be a unique migration name.)*

Then verify:

```bash
npx prisma migrate status
```

## Notes for future migrations

Once these catch-up files are applied, switch to the normal Prisma workflow:

```bash
# After any schema change:
npx prisma migrate dev --name describe_your_change
# This creates a timestamped migration in prisma/migrations/ and applies it.
```

Do NOT use `prisma db push` on production going forward.
